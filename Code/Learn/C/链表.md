学弟，链表（Linked List）是 C 语言逻辑构建的“巅峰”，也是你从管理“静态空间”转向构建“动态世界”的标志。

如果说可变数组是“搬家”，那链表就是“寻宝游戏”：**每一个节点（Node）都藏着一份宝藏（Value）和一张去往下一个节点的地图（Next 指针）。**

为了配合你的学习节奏，我将链表分为**三个部分**讲解。现在开始第一部分：**结构定义与动态构建**。

# 链表专题（一）：逻辑原子与动态增长

## 1. 节点的解剖：自引用结构
链表的基本单位是**节点（Node）**。在 C 语言中，它必须通过“自引用结构体”来实现。

```c
typedef struct _node {
    int value;           // 数据域：存放实际值
    struct _node *next;  // 指针域：存放下一个节点的地址
} Node;
```

### 💡 [极客洞察] 为什么这里要写 `struct _node *next`？
在执行 `typedef` 之前，编译器还不认识 `Node` 这个名字。为了让结构体内部能指向同类型的自己，必须使用原始的标签名 `struct _node`。

## 2. 链表的容器化管理
虽然可以直接操作 `head` 指针，但翁恺老师推荐使用一个专门的 `List` 结构体来包裹头指针。

```c
typedef struct _list {
    Node* head;
    // Node* tail; // 进阶：记录尾指针可将插入效率从 O(n) 提升至 O(1)
} List;
```
*   **优势**：这种“容器化”设计将“链表本身”与“链表的节点”分离。当你需要修改头指针时，通过传递 `List *` 指针，可以避免复杂的二级指针（Pointer to Pointer）操作。

## 3. 动态构建：尾部插入算法 (Append)
这是你在第一张截图里看到的逻辑。我们要实现一个方案：**每输入一个数，就新建一个节点挂在链表最末尾。**

### 方案设计步骤：
1.  **分配空间**：使用 `malloc` 为新节点申请内存并初始化（`next = NULL`）。
2.  **定位边界**：
    *   如果链表是空的（`head == NULL`），新节点就是 `head`。
    *   如果链表不空，必须**顺藤摸瓜**找到最后一个节点。
3.  **连接**：将原末尾节点的 `next` 指向新节点。

### 代码实现逻辑：
```c
void add(List* pList, int number) {
    // 1. 准备新节点
    Node *p = (Node*)malloc(sizeof(Node));
    p->value = number;
    p->next = NULL;

    // 2. 寻找尾部
    Node *last = pList->head;
    if (last) {
        while (last->next) { // 只要当前节点的下一个不为空，就往后走
            last = last->next;
        }
        // 3. 挂接
        last->next = p;
    } else {
        // 特殊情况：处理空链表
        pList->head = p;
    }
}
```

---

## 💡 [NJU ICS/PA 深度补丁]

1.  **$O(n)$ 的代价**：注意看那个 `while` 循环。为了找到末尾，你每次都要从头遍历。这意味着插入第 $n$ 个元素的代价是 $n$ 次查找。在大型系统中，我们通常会维护一个 `tail` 指针来消除这个循环。
2.  **内存对齐的再思考**：`sizeof(Node)` 是多少？
    *   在 32 位环境下：`int(4)` + `next(4)` = 8 字节。
    *   在 64 位环境下：`int(4)` + `Padding(4)` + `next(8)` = 16 字节。
3.  **NULL 的警觉**：永远记住，`last->next` 能够执行的前提是 `last` 本身不是 `NULL`。这就是为什么代码里必须先有一个 `if (last)` 的判断。

# 链表专题（二）：遍历、检索与“拆楼”逻辑

## 1. 遍历 (Traversal)：顺着藤摸瓜
遍历是链表一切操作的基础。由于链表节点在内存中是“散装”的，我们只能靠一个移动的指针（通常叫 `p`）去挨个访问。

### 方案设计：
1. **设置游标**：定义一个 `Node *p`，初始指向 `head`。
2. **循环推进**：只要 `p` 不是 `NULL`，就处理当前节点，并执行 `p = p->next`。
3. **终止条件**：当 `p` 变成 `NULL` 时，说明走到了尽头。

### 标准代码模板：
```c
for (Node *p = list.head; p != NULL; p = p->next) {
    printf("%d\t", p->value);
}
```
> **[NJU 避坑补丁]**：注意区分 `while(p)` 和 `while(p->next)`。
> * `while(p)`：你会进入每一个房间，包括最后一个。
> * `while(p->next)`：你会停在**最后一个房间**里，不进去那个“虚无的 NULL”。

---

## 2. 搜索 (Search)：大海捞针
搜索本质上就是“带条件的遍历”。

### 架构师方案：
* **输入**：目标数值 `target`。
* **逻辑**：遍历链表，比对 `p->value == target`。
* **返回**：如果找到，返回该节点的**地址**；如果走到底都没找到，返回 `NULL`。

---

## 3. 删除 (Delete)：逻辑的修剪艺术
这是链表最难、也是最容易让程序“崩盘”的操作。

### 方案设计（核心思维）：
想象一列火车，你要拿掉其中的一节车厢。你不能直接把它抽走（否则后面的车厢就丢了），你必须**先把前面的挂钩，连到后面的车钩上**。

#### 第一步：双指针法 (The Two-Pointer Chase)
你需要两个指针：
* `p` (Current)：当前正在看、准备杀掉的节点。
* `q` (Previous)：`p` 的前一个节点，负责“善后”。

#### 第二步：边界判定 (Boundary Check)
根据你要删的点的位置，分两种情况：
1. **删的是“头节点”**：没有 `q`。直接让 `list.head` 指向 `head->next`。
2. **删的是“中间/尾节点”**：让 `q->next` 跳过 `p`，指向 `p->next`。

#### 第三步：物理销毁 (Memory Cleanup)
**绝对不能忘记 `free(p)`**！否则这块内存就成了死在内存里的“僵尸”。

### 截图 4 中的“职业写法”拆解：
```c
Node *p, *q;
for (q = NULL, p = list.head; p; q = p, p = p->next) {
    if (p->value == target) {
        if (q) {
            q->next = p->next; // 绕过 p
        } else {
            list.head = p->next; // 删的是头
        }
        free(p); // 物理删除
        break;   // 删完收工
    }
}
```

---

## 💡 [NJU ICS/PA 深度补丁]

1. **“死后不能说话”**：
   千万不要先写 `free(p)` 再写 `q->next = p->next`。一旦 `free` 了，`p->next` 所在的地址就已经不合法了。这是一个非常经典的 **Use-After-Free** 错误。

2. **指针的“左值”警觉**：
   当你写出 `p->next` 时，你的大脑必须瞬间反应：**“万一 p 是 NULL 呢？”**。在南大的代码审计中，如果不判空就直接 `->`，是直接打回重写的。

3. **内存视图的变化**：
   删除节点后，哈斯图或内存链条会缩短。建议你在调试时，打印出 `p`、`q`、`p->next` 的**十六进制地址**，看着那些地址是如何重新连接的。

学弟，你这一版代码的**整体框架和方案**已经非常成型了！特别是 `list_delete` 中双指针（$p, q$）的配合，逻辑走得非常准。

但是，作为一个正在培养“审计直觉”的 NJUer，我们要对代码里的**三处严重 Bug** 进行一次“捉虫实验”。这三处错误正是系统编程中最容易让程序“莫名失踪”或“内存腐烂”的地方。

### 🔍 代码审计：找出隐藏的威胁

#### 1. `list_search`：返回地址的陷阱
看你的这一段：
```c
if (p->value == target) {
    return &p;  // ❌ 致命错误
}
```
*   **为什么错了？** 
    *   `p` 是一个局部变量（指针），它存的是某个节点的地址（比如 `0x100`）。
    *   `&p` 取的是**这个指针变量自己在栈上的地址**（比如 `0x7fff...`）。
    *   一旦函数结束，变量 `p` 就在栈上消失了，你返回的 `&p` 就成了一个**悬空指针**。
*   **修正**：你想要的是返回那个节点的地址，所以直接 `return p;` 即可。
*   **漏掉的逻辑**：如果循环结束还没找到，你应该返回一个 `NULL` 告诉调用者没找到。

#### 2. `list_delete`：括号的大危机
这是你这段代码里最隐蔽的逻辑错误（注意你的大括号范围）：
```c
if (p->value == target)
{
    if (q) { q->next = p->next; }
    else { plist->head = p->next; }
}
free(p);  // ❌ 错误位置
break;   // ❌ 错误位置
```
*   **后果模拟**：不管 `p->value` 是不是 `target`，程序运行到这里都会执行 `free(p)` 和 `break`。
*   **结果**：你这段代码不论搜什么，都只会**删掉链表的第一个节点，然后强行结束**。
*   **修正**：`free(p)` 和 `break` **必须**放在 `if (p->value == target)` 的大括号里面。只有找到了目标，才需要“销毁”和“跳出”。

---

### 🛠️ 职业选手版：修复后的实现

我根据你的代码进行了针对性的微调，请对比差异：

```c
Node *list_search(List *plist, int target)
{
    for (Node *p = plist->head; p; p = p->next)
    {
        if (p->value == target)
        {
            return p; // ✅ 直接返回指针 p 存储的地址
        }
    }
    return NULL; // ✅ 明确告知没找到
}

void list_delete(List *plist, int target)
{
    Node *p, *q;
    for (p = plist->head, q = NULL; p; q = p, p = p->next)
    {
        if (p->value == target)
        {
            if (q) {
                q->next = p->next;
            } else {
                plist->head = p->next;
            }
            free(p); // ✅ 只有找到了才 free
            return;  // ✅ 删完直接退出函数（或者 break）
        }
    }
}
```

### 💡 链表大结局：如何彻底“拆楼”？ (Clear All)

当你程序结束时，仅仅 `free` 一个节点是不够的。你需要一个能把整栋楼拆干净的方案。

#### 方案设计（Clear List）：
1. 站在门口（`head`）。
2. 在拆掉当前房间之前，**必须先拿走去下一个房间的纸条**（备份 `next` 指针）。
3. 拆房（`free`）。
4. 顺着纸条去下一个房间，直到无房可拆。

```c
void list_clear(List *plist) {
    Node *p = plist->head;
    while (p) {
        Node *next = p->next; // 1. 先记下后路
        free(p);             // 2. 拆掉当前节点
        p = next;            // 3. 走向下一节点
    }
    plist->head = NULL;      // 4. 将头设为空，防止二次访问
}
```
