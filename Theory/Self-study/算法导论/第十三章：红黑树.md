# 《算法导论》第十三章：红黑树全章笔记

## 1. 红黑树的本质与定义
红黑树是一种**自平衡**的二叉搜索树。它保证没有任何一条路径会比其他路径长出两倍，因此在最坏情况下，基本动态集合操作的时间复杂度均为 **$O(\lg n)$**。

### 红黑性质：
1.  **每个节点或者是红色的，或者是黑色的。**
2.  **根节点是黑色的。**
3.  **每个叶节点（NIL）是黑色的。**
4.  **如果一个节点是红色的，则它的两个子节点都是黑色的。**（不能有两个连续的红节点）
5.  **对每个节点，从该节点到其所有后代叶节点的简单路径上，均包含相同数目的黑色节点。**（黑高相等）

---

## 2. 哨兵节点 (Sentinel `T.nil`)

在写代码实现时，如果给每个叶子都弄一个真实的 NULL 节点，太浪费空间且容易写错。

* **天才的想法**：用一个统一的哨兵对象 `T.nil` 来代表所有的叶节点
* **属性**：它的 color 是 **BLACK**，其他属性（key, left, right）可以设为任意值。
* **好处**：这大大简化了插入和删除时的边界判断。你可以像对待普通节点一样对待 `T.nil`，不用每次都判断 `if (x != NULL)`。

## 3. 旋转（Rotations）：结构调整的原子操作

当我们在红黑树中进行插入或删除时，性质可能会被破坏。**旋转**是用来修复性质且保持 BST 性质（左小右大）的手段。

### 左旋 (Left-Rotate) 与 右旋 (Right-Rotate)
*   **左旋逻辑**：将节点 $x$ 向左下移动，让它的右孩子 $y$ 变成新的根，同时 $y$ 的左子树变成 $x$ 的右子树。
*   **特性**：旋转操作在 **$O(1)$** 时间内完成，且**不改变中序遍历的结果**。
*   左旋 $x$：让 $x$ 的右孩子 $y$ 上升，让 $x$ 下降成为 $y$ 的左孩子，同时把 $y$ 原来的左子树 $\beta$ 挂到 $x$ 的右边

**C++ 代码实现（左旋）：**

```cpp
void left_rotate(Node* &root, Node* x, Node* nil) {
    Node* y = x->right;         // 设置 y
    x->right = y->left;         // 将 y 的左子树转为 x 的右子树
    if (y->left != nil)
        y->left->p = x;
    y->p = x->p;                // 将 x 的父节点连到 y 上
    if (x->p == nil)
        root = y;
    else if (x == x->p->left)
        x->p->left = y;
    else
        x->p->right = y;
    y->left = x;                // 把 x 放回 y 的左边
    x->p = y;
}
```

## 4. 插入 (Insertion)
插入一个新节点分为两步：
1.  **按 BST 逻辑插入**：将新节点设为**红色**（为了不破坏黑高性质 5）。
2.  **修复 (RB-INSERT-FIXUP)**：如果新节点的父亲也是红色，违反了性质 4，则需要通过变色和旋转来修复。

### 插入修复的三种情况（核心在于看“叔叔”的颜色）：
*   **Case 1**：父叔同红：父叔变黑，祖父变红，问题上移。
*   **Case 2**：叔黑且内侧：先旋转父结点，把它转成情况 3。
*   **Case 3**：叔黑且外侧：父变黑，祖父变红，对祖父旋转，完成修复。

```text
父亲是红色（当父亲为祖父的左孩子时，如果是右孩子对称）
    → 看叔叔颜色
        → 叔叔红：父叔黑，祖父红，继续向上成为祖父
        → 叔叔黑：看 z 是左孩子还是右孩子
        	→左孩子：左旋父节点
        	→右孩子：父变黑，祖父变红，右旋祖父
```

## 5. 删除 (Deletion) —— 最终挑战
先把 **(z,y,x)** 和 **额外黑色**讲清楚。

---

# 1. 先说三个角色：(z,y,x)

在 `RB-DELETE(T, z)` 里：

```text
z：你“想删除”的那个结点
y：实际被删除或被移动的那个结点
x：接替 y 原来位置的那个结点
```

这三个角色一定要分开。

---

## 情况一：(z) 没有左孩子

比如：

```text
    z
     \
      x
```

删除 (z) 后，让 (x) 接替 (z) 的位置。

这时：

```text
y = z
x = z.right
```

也就是说：

```text
z：目标删除结点
y：实际被删掉的结点，就是 z
x：顶上来接替 z 的结点
```

---

## 情况二：(z) 没有右孩子

比如：

```text
      z
     /
    x
```

删除 (z) 后，让 (x) 接替 (z)。

这时：

```text
y = z
x = z.left
```

也还是：

```text
y = z
```

---

## 情况三：(z) 有两个孩子

比如：

```text
        z
       / \
      A   ...
           \
            ...
```

普通二叉搜索树删除时，不能直接把 (z) 摘掉，因为 (z) 左右都有子树。

所以我们找 (z) 的后继 (y)，也就是 (z) 右子树里最小的结点。

例如：

```text
        z
       / \
      A   R
         /
        y
         \
          x
```

删除时，实际操作是：

1. 把后继 (y) 从原来的位置拿出来；
2. 把 (y) 移到 (z) 的位置；
3. (x) 接替 (y) 原来的位置。

这时：

```text
z：你想删除的目标结点
y：z 的后继，它会被移动到 z 的位置
x：接替 y 原来位置的结点
```

所以这时真正“从原位置消失”的不是 (z)，而是 (y)。

这就是为什么算法一直关心：

```text
y-original-color
```

而不是：

```text
z-original-color
```

因为红黑性质是否被破坏，关键取决于**从原位置拿走的那个结点 (y) 原来是什么颜色**。

---

# 2. 为什么关心 (y) 的颜色？

删除红色结点，一般不麻烦。

因为红色结点不计入黑高。

例如删除一个红色叶子：

```text
      P
     /
   y(R)
```

删掉 (y) 后，从 (P) 到这边叶子的路径上，黑色结点数量没有减少。

所以如果：

```text
y-original-color = RED
```

通常不用修复。

---

但是如果 (y) 原来是黑色，麻烦就来了。

因为黑色结点是计入黑高的。

比如原来某条路径是：

```text
... -> y(B) -> ...
```

现在把 (y) 拿走了，这条路径就少了一个黑色结点。

于是同一个祖先到不同叶子的路径上，黑色结点数量可能不一样了。

这就违反了红黑树性质 5：

> 从任意结点到其所有后代叶子的简单路径上，黑结点数量相同。

所以：

```text
删除红色 y：一般安全
删除黑色 y：可能导致某些路径少一个黑色
```

---

# 3. 那 (x) 是什么？

(x) 是接替 (y) 原来位置的结点。

如果 (y) 被拿走了，树上原来属于 (y) 的位置必须有人顶上。

这个顶上来的就是 (x)。

例如：

```text
      P
     /
   y(B)
     \
      x
```

删除 (y) 后：

```text
      P
     /
    x
```

所以 (x) 所在的这边路径，原来经过 (y(B))，现在不经过了。

因此，从 (P) 往 (x) 这边走的路径少了一个黑色结点。

---

# 4. “额外黑色”到底是什么意思？

它不是说 (x) 真的多了一个颜色。

它是一种**记账方法**。

删除黑色 (y) 之后，经过 (x) 的那些路径少了一个黑色结点。我们有两种说法。

---

## 说法一：路径少了一个黑色

删除前：

```text
      P
     /
   y(B)
     \
      x
```

经过这边的路径包含 (y(B))。

删除后：

```text
      P
     /
    x
```

现在 (y(B)) 没了，所以这边路径少一个黑色。

---

## 说法二：把 (y) 的黑色“推给” (x)

为了方便修复，算法导论换了一个视角：

> 虽然 (y) 被删了，但我们先假装它的那一层黑色没有消失，而是转移到了 (x) 身上。

也就是说，(x) 现在除了自己的颜色之外，还额外背着一层黑色。

这就是：

```text
extra black
```

也就是“额外黑色”。

---

# 5. 举个最简单的例子

假设删除前：

```text
      P(B)
     /   \
   y(B)  S(B)
```

其中 (y) 是黑色叶子。

删除 (y) 后，左边变成 (T.nil)。

普通看法：

```text
      P(B)
     /   \
  T.nil  S(B)
```

现在左边路径少了一个黑色。

为了不说“左边少黑”，我们说：

```text
      P(B)
     /   \
  x(BB)  S(B)
```

这里 (x = T.nil)，但是它背着额外一层黑色，所以可以理解成：

```text
x = 黑 + 黑
```

这就是“双重黑色”。

---

# 6. 为什么这样记账有用？

因为这样一来，我们可以暂时认为性质 5 还成立。

原来左边少了一个黑色，但如果我们说 (x) 多带一个黑色，那么路径黑色数又补回来了。

也就是说：

```text
真实情况：经过 x 的路径少一个黑色

记账视角：让 x 多背一个黑色，所以路径黑色数暂时平衡
```

但是问题转移成了：

```text
x 现在不是正常的红色或黑色，而是带着额外黑色
```

这违反了性质 1：

> 每个结点要么是红色，要么是黑色。

所以 `RB-DELETE-FIXUP` 的任务就是：

```text
消除 x 身上的额外黑色
```

---

# 7. 如果 (x) 原本是红色呢？

这很好处理。

如果 (x) 原来是红色，又背了一层额外黑色，那么它是：

```text
红 + 黑
```

算法最后可以直接把它染成黑色：

```text
x.color = BLACK
```

这样：

```text
红 + 黑  →  黑
```

额外黑色被吸收了，问题结束。

这对应正文说的：

```text
x points to a red-and-black node
```

也就是 (x) 是“红黑”结点。

---

# 8. 如果 (x) 原本是黑色呢？

如果 (x) 原来就是黑色，又背了一层额外黑色，那么它是：

```text
黑 + 黑
```

也就是“双重黑色”。

这种最麻烦。

`while` 循环：

```text
while x != T.root and x.color == BLACK
```

可以理解为：

> 只要 (x) 不是根，并且 (x) 本身是黑色，那么它现在就是“双黑”，需要继续修复。

注意这里代码里没有写：

```text
x.color == DOUBLE_BLACK
```

因为“双黑”不是实际存储的颜色。

它只是算法分析中的概念。

代码只用普通的 `BLACK` 表示 (x) 本身是黑色，而“额外黑色”是我们脑子里跟踪的东西。

---

# 9. 删除修复的本质

`RB-DELETE-FIXUP` 的四种情况，其实都在做一件事：

```text
处理 x 身上的额外黑色
```

有三种结局：

---

## 结局一：被红色 (x) 吸收

如果 (x) 是红色加额外黑色：

```text
红 + 黑
```

最后染成黑色即可。

```text
x.color = BLACK
```

---

## 结局二：额外黑色传到根

如果 (x) 一路上移到了根：

```text
x == T.root
```

那么额外黑色可以直接去掉。

因为整棵树所有路径都经过根，从根处消掉额外黑色，不会造成左右路径不平衡。

---

## 结局三：通过兄弟结点旋转和变色消掉

也就是 case 4：

通过改变 (x.p)、兄弟 (w)、侄子结点的颜色，再旋转，把多出来的黑色重新分配掉。

---

# 10. 你可以这样记 (z,y,x)

非常重要的一句话：

```text
z 是用户要删的结点；
y 是真正被“拿走颜色影响”的结点；
x 是顶替 y 原位置、背锅的结点。
```

更形象一点：

```text
z：我要删除谁？
y：实际从原位置消失的是谁？
x：谁顶上去，并承担少掉的黑色？
```

---

# 11. 为什么不是 (z) 背额外黑色，而是 (x)？

因为红黑性质出问题的地方，不一定在 (z) 原来的位置。

如果 (z) 有两个孩子，我们不是简单摘掉 (z)，而是把后继 (y) 移到 (z) 的位置。

真正被挖空的位置是 (y) 原来的位置。

而接替这个空位的是 (x)。

所以黑高出问题发生在：

```text
y 原来的位置
```

不是：

```text
z 原来的位置
```

因此额外黑色交给：

```text
x
```

---

# 12. 用一句话把删除过程串起来

红黑树删除可以这样理解：

```text
先按 BST 删除的逻辑处理 z。

但真正影响黑高的是被删除或被移动的 y。

如果 y 原来是红色，没事。

如果 y 原来是黑色，那么 y 原来位置上的路径少了一个黑色。

于是让接替 y 原位置的 x 背上一层“额外黑色”。

然后调用 RB-DELETE-FIXUP(T, x)，想办法把这层额外黑色消掉。
```

你现在先抓住这个，后面四种情况就会清晰很多：

```text
删除修复不是在修 z，
而是在修 x 身上从 y 那里继承来的额外黑色。
```
## 删除逻辑链
```text
while循环条件：只要x不是根并且x为黑色（依旧对称，所以只考虑x为它父亲的左孩子的情况）
Case 1：兄弟红
    通过变色 + 旋转，把它转成兄弟黑的情况。

Case 2：兄弟黑，两个侄子都黑
    兄弟变红，额外黑色上移到父亲。

Case 3：兄弟黑，近侄红，远侄黑
    旋转兄弟，转成 Case 4。

Case 4：兄弟黑，远侄红
    旋转父亲，重新染色，直接消除额外黑色。
```

## 6. 💡 助教的深度总结与感悟

### 为什么红黑树比普通 BST 强大？
*   **稳定的效率**：在实际应用中，数据往往不是完全随机的。如果数据是接近有序的，普通 BST 会非常慢，而红黑树通过强制性的“五条性质”，保证了无论数据多烂，它都能在大约 $2\lg n$ 的高度内完成任务。

### 为什么是红色和黑色？
*   其实颜色只是一个标记。红黑树的本质是模拟 **2-3-4 树**。红色节点可以看作是与其父节点合并在一起形成一个大的节点。通过颜色，我们用二叉树的形式实现了一棵极其平衡的多路搜索树。
